# API 연동 가능성 검토

작성일: 2026-05-08

## 결론

현재 기획한 4개 API는 모두 무료 또는 무료 구간이 있지만, 실제 서비스 수준으로 호출하려면 인증키, 활용신청, 애플리케이션 등록이 필요하다.

즉, 별도의 키 발급 없이 바로 운영 데이터로 끌어오는 방식은 어렵다. 개발 단계에서는 mock 데이터로 UI와 사용자 흐름을 먼저 만들고, 이후 API adapter만 교체하는 구조가 현실적이다.

## API별 판단

| API | 키 없이 바로 호출 | 필요한 절차 | 비고 |
|---|---:|---|---|
| 서울 실시간 도시데이터 | 제한적 | 서울열린데이터광장 인증키 신청 | 샘플키는 일부 장소만 가능 |
| 서울교통공사 승하차 인원 | 제한적 | 공공데이터포털 활용신청/인증키 | 파일 다운로드는 가능하나 실시간 proxy로 쓰기엔 한계 |
| 기상청 단기예보 | 불가 | 공공데이터포털 활용신청/ServiceKey | 요청 파라미터에 ServiceKey 필수 |
| ODsay 대중교통 길찾기 | 불가 | ODsay 애플리케이션 등록/API Key | Basic 무료 6개월, 1,000회/일 |

## 제품 설계 방향

1. 화면은 mock 데이터로 먼저 완성한다.
2. 데이터 소스마다 adapter를 둔다.
3. API 키는 앱에 직접 넣지 않고 백엔드 프록시에서 관리한다.
4. 사용자가 보는 UI에는 “공공데이터 지연값 + 길라미 AI 보정값”처럼 신뢰 가능한 문구로 표현한다.
5. 서울교통공사 승하차 데이터는 진짜 T-0 실시간 데이터라기보다 시간대별 패턴 보정용으로 보는 것이 안전하다.

## UX 원칙

- 외국인 관광객에게 “API 연결 실패” 같은 개발자 언어를 보여주지 않는다.
- 대신 “현재 공공데이터 지연값을 AI가 보정 중”처럼 서비스 가치 중심으로 보여준다.
- 데이터 신뢰도, 업데이트 시각, 예측 근거를 함께 보여준다.
- 경고는 공포를 주기보다 바로 할 수 있는 행동을 제안한다.

## 실제 구동 방법

현재 프로젝트는 mock 데이터만으로 동작한다.

```bash
cd seoul-nowcast
npm install
npm run web
```

웹 실행 후 Expo가 안내하는 주소 또는 로컬 주소로 접속한다.

```text
http://localhost:8081
```

## 추후 실제 API 전환 순서

1. 서울열린데이터광장 인증키 발급
2. 공공데이터포털 계정 생성 및 기상청 단기예보 활용신청
3. ODsay LAB 애플리케이션 등록 및 API Key 발급
4. 백엔드 프록시에서 각 API 호출
5. 앱은 백엔드의 정규화된 응답만 사용
6. mock 데이터와 실제 응답 스키마를 맞춰 단계적으로 교체
