## 1. 문제 정의 (Problem Statement)

### 1-1. Why — 왜 이 문제를 풀어야 하는가?

2026년 5월 1일, 서울 성수동 포켓몬 30주년 행사에 약 16만 명이 몰리며 인파 마비 사태가 발생했다. 경찰이 투입되어 행사가 중단되었고, 통신 장애까지 발생했다. 이 사고의 핵심 원인은 **"사람들이 현장에 도착하기 전, 그 장소가 얼마나 위험한지 알 수 있는 방법이 없었다"**는 것이다.

서울시는 이미 '실시간 도시데이터'를 제공하고 있지만, 아래 **3가지 치명적인 한계(Pain Point)**로 인해 사고 예방에 실질적으로 기여하지 못하고 있다.

### 1-2. Pain Points (핵심 고통점)

| # | Pain Point | 설명 | 심각도 |
|---|-----------|------|--------|
| P1 | **30분 지연된 데이터** | 통신사 기지국 데이터 집계에 약 30분 소요. 오후 2시 30분에 보는 데이터는 2시 정각 기준. 급변하는 인파에 무용지물. | 🔴 Critical |
| P2 | **미래 예측 부재** | 현재 상태(여유/보통/붐빔)만 표시. "내가 도착할 시점"의 상황은 알 수 없음. | 🟠 High |
| P3 | **솔루션 부재** | 혼잡하다는 사실만 알릴 뿐, "그래서 언제 가야 하는가?" 등 사용자 행동 변화를 유도하는 대안이 없음. | 🟡 Medium |

## 2. Vision Statement (비전)

**"AI로 `[ 공공데이터의 30분 지연(Blind Spot) ]`을 없앨 수 있다면, 우리는 군중 밀집 사고를 가장 선제적으로 막을 수 있다."**

**슬로건**: "서울시 데이터는 30분 느리지만, AI는 30분 앞서갑니다."

## 3. Target Users (타겟 유저 & 페르소나)

### 페르소나 1: 김서연 (Primary)
- **나이/직업**: 26세 / 대학원생
- **거주지**: 서울 마포구
- **행동 패턴**: 주말마다 홍대, 성수동 팝업스토어 방문. SNS에서 핫한 장소 정보를 수집.
- **Pain**: "토요일 오후에 홍대 가려고 했는데, 도착하니 사람이 너무 많아서 줄만 서다 돌아왔어요. 가기 전에 알았으면 시간을 아꼈을 텐데."
- **Goal**: 목적지의 혼잡도를 사전에 파악하여, 쾌적한 시간대에 방문하고 싶다.

### 페르소나 2: 이준호 (Secondary)
- **나이/직업**: 34세 / 직장인 (판교 근무, 아이 1명)
- **거주지**: 서울 강남구
- **행동 패턴**: 주말에 가족과 나들이(여의도 한강공원, 잠실 등). 인파에 대한 불안감이 큼.
- **Pain**: "이태원 사고 이후로 사람 많은 곳에 아이를 데리고 가는 게 무서워요. 안전한 시간대를 알고 싶어요."
- **Goal**: 가족의 안전을 확보할 수 있는 안전한 시간대와 장소 정보를 원한다.

### 페르소나 3: 성동구청 안전과 담당자 (Tertiary / B2G)
- **Pain**: "행사 개최 시 실시간 인파 모니터링 도구가 없어 경찰 투입 시점을 놓쳐 사고가 커진다."
- **Goal**: 임계치 초과 시 자동 경보 시스템이 필요하다.

## 4. Value Proposition (가치 제안 — JTBD 6-Part)

| 항목 | 내용 |
|------|------|
| **Who** (누가) | 주말/공휴일에 서울 핫플레이스를 방문하려는 20~40대 |
| **Why** (왜 필요) | 도착 시점의 혼잡도를 알 수 없어 헛수고·시간 낭비·안전 위험에 노출 |
| **What Before** (기존 방법) | 서울시 실시간 데이터 직접 확인 → 30분 지연 데이터 → 의미 없음 |
| **How** (우리의 해결책) | AI가 지연 데이터 + 실시간 대중교통 보조 데이터를 융합하여 현재~30분 뒤 혼잡도를 예측하고, 최적 방문 시간대를 넛지 |
| **What After** (사용 후) | 쾌적한 시간대에 안전하게 방문. 시간 낭비 최소화. |
| **Alternatives** (대안) | 서울시 데이터(30분 지연), 네이버 지도 혼잡도(구 단위·비정량적), 직접 전화 |

## 5. 핵심 기능 (Core Feature — One Thing)

### 🚀 지연 보정 초단기 예측 엔진 (Nowcasting ML Engine)

**단 하나의 핵심 기능에만 집중합니다.**

1. **Input**: 서울시 30분 지연 인구 데이터(T-30) + 실시간 지하철 하차 인원(T-0) + 실시간 날씨
2. **AI Processing**: 시계열 ML(XGBoost)이 현재(T-0) 인구를 추론하고, 30분 뒤(T+30) 인구를 예측
3. **Output**: 시간축 그래프 (현재 → 도착 예정 시간의 혼잡도 변화) + 최적 방문 시간 넛지

**사용자 시나리오 (홍대입구 기준)**:
> 토요일 오후 1시, 김서연이 앱을 열고 '홍대입구'를 선택합니다.
> 
> 화면에 시간축 그래프가 나타납니다:
> - 현재(13:00): 🟢 여유 (약 3.2만 명)
> - 14:00 (도착 예상): 🟠 약간 붐빔 (약 4.8만 명)
> - 15:00: 🔴 매우 혼잡 (약 6.5만 명 예측)
> - 17:00: 🟡 보통 (약 4.1만 명 예측)
>
> AI 넛지: "💡 15시 피크를 피해 17시 이후 방문을 추천합니다. 예상 혼잡도가 37% 감소합니다."

## 6. Risky Assumptions (검증이 필요한 가설)

| # | 가설 | 리스크 | 검증 방법 |
|---|------|--------|----------|
| A1 | 30분 지연된 인구 데이터와 실시간 지하철 하차 데이터 사이에 유의미한 상관관계가 존재한다 | 🔴 High | 과거 데이터 회귀 분석(백테스팅) |
| A2 | 사용자는 AI 예측 결과를 신뢰하고 출발 시간을 실제로 변경할 것이다 | 🔴 High | 프로토타입 사용성 테스트, 설문조사 |
| A3 | 시계열 ML(XGBoost)이 인파 급증 상황에서도 오차율 15% 미만으로 예측할 수 있다 | 🟠 Medium | 성수동 포켓몬 사태 당일 데이터로 스트레스 테스트 |

## 7. North Star Metric (핵심 성과 지표)

| 지표 | 정의 | 초기 목표 |
|------|------|----------|
| **🌟 Time-Shift Actions** | 사용자가 AI 예측을 확인한 뒤, 출발 시간을 변경(지연/앞당김)한 횟수 | 주간 1,000회 |
| Input Metric 1: 예측 정확도 | AI 예측값 vs 실제 발표값의 오차율(MAPE) | < 15% |
| Input Metric 2: DAU | 일간 활성 사용자 수 | 5,000명 |
| Input Metric 3: 넛지 수용률 | AI 추천 시간대를 확인한 유저 중 실제 시간을 바꾼 비율 | > 30% |
