# 최종 기획서: 널널

> **주제: AI가 헛걸음을 없앤다면, 서울 여행은 더 널널해질 수 있다.**

서비스명: **널널**  
영문명: **NULL NULL**  
AI 비서명: **길라미 AI**  
핵심 타겟: **서울을 방문하는 외국인 관광객**  
확장 타겟: 서울 핫플레이스 방문자, 팝업스토어 방문자, 서울 거주자, 가족 단위 방문객

---

## 1. Executive Summary

**널널**은 서울의 실시간 혼잡도, 날씨, 대중교통, 장소 정보를 바탕으로 사용자가 “지금 가도 되는지”, “언제 가면 덜 붐비는지”, “어디로 바꾸면 좋은지”를 판단하도록 돕는 AI 여행 비서 서비스다.

기존 지도 앱은 빠른 길과 장소 검색에는 강하지만, 현장의 혼잡도와 여행 실패 가능성까지 고려해 행동을 제안하지는 못한다. 널널은 단순히 “사람이 많다”를 보여주는 서비스가 아니라, **AI가 혼잡 상황을 해석하고 사용자가 바로 실행할 수 있는 선택지**를 제공하는 서비스다.

초기 MVP는 외국인 관광객이 서울 주요 핫플레이스 방문 전 혼잡도와 대체 장소를 확인하도록 돕는 데 집중한다.

---

## 2. 제품 비전

### Vision

서울을 방문하는 사람이 혼잡 때문에 시간을 낭비하거나 여행 계획을 망치지 않도록, AI가 더 안전하고 여유로운 선택을 제안한다.

### Mission

실시간 도시 데이터와 AI 분석을 활용해 사용자가 서울에서 **덜 기다리고, 덜 헤매고, 더 만족스럽게 이동**하도록 돕는다.

### Product Promise

> 널널은 “지금 가도 될까?”라는 질문에 가장 빠르고 행동 가능한 답을 준다.

---

## 3. 문제 배경

서울은 홍대, 성수, 명동, 강남, 이태원, 경복궁, 한강공원처럼 사람들이 몰리는 장소가 많다. 특히 SNS, 팝업스토어, K-pop, 맛집, 카페 문화가 결합되면서 특정 장소에 갑자기 사람이 몰리는 일이 자주 발생한다.

사용자는 출발 전에는 장소가 얼마나 붐비는지 정확히 알기 어렵다. 대부분 도착한 뒤에야 줄이 길거나, 자리가 없거나, 이동이 불편하다는 사실을 알게 된다.

외국인 관광객은 이 문제가 더 크다. 한국어 안내, 지하철 출구, 교통카드, 운영시간, 결제 방식, 문화 차이까지 함께 고려해야 하기 때문이다.

---

## 4. 사용자 조사 기반 Pain Points

### Pain Point 1. 도착 후에야 혼잡을 알게 된다

사용자는 핫플레이스, 팝업스토어, 맛집, 카페에 도착한 뒤에야 줄이 길거나 자리가 없다는 사실을 알게 된다.

대표 상황:

- 팝업스토어에 갔지만 줄이 너무 길어서 포기한다.
- 음식점에 사람이 너무 많아 옆집으로 이동한다.
- 유명 장소에 가려다 사람이 많아서 못 간다.
- 카페나 명소에 자리가 없어 원래 계획을 포기한다.

정리 문장:

> 사용자는 목적지의 혼잡도를 사전에 알 수 없어, 도착 후에야 계획을 수정하게 되고 여행 시간과 만족도를 잃는다.

### Pain Point 2. 혼잡과 날씨 상황에서 대안을 찾기 어렵다

사람이 많거나 비가 올 때 사용자는 어디로 바꿔야 할지 모른다. 결국 아무 곳에 들어가거나 숙소로 돌아가는 등 시간이 낭비된다.

대표 상황:

- 홍대가 붐비는데 비슷한 분위기의 대체 장소를 모른다.
- 비가 오는데 가까운 실내 관광지를 빠르게 찾기 어렵다.
- 대기 시간이 길어졌지만 근처에서 기다릴 장소를 모른다.
- 외국인은 검색 결과가 한국어 중심이라 판단이 더 어렵다.

정리 문장:

> 사용자는 혼잡이나 날씨로 계획이 틀어졌을 때, 지금 이동 가능한 대체 장소를 빠르게 찾기 어렵다.

### Pain Point 3. 숫자보다 행동 가능한 안내가 필요하다

사용자는 “현재 5만 명” 같은 숫자만으로는 상황을 판단하기 어렵다. 중요한 것은 사람이 몇 명인지가 아니라, 지금 가야 하는지, 기다려야 하는지, 다른 곳으로 가야 하는지다.

사용자가 원하는 안내:

- 지금 가지 마세요.
- 30분 뒤에 가세요.
- 성수는 지금 여유로워요.
- 홍대 대신 을지로를 추천해요.
- 9번 출구 대신 2번 출구로 이동하세요.

정리 문장:

> 사용자는 혼잡도 수치 자체보다, 현재 상황에서 어떤 선택을 해야 하는지 알려주는 행동 중심 안내를 필요로 한다.

### Pain Point 4. 외국인 관광객은 추가 장벽을 겪는다

외국인 관광객은 혼잡 문제뿐 아니라 언어, 대중교통, 결제, 운영시간, 문화 차이 때문에 장소 선택과 이동 판단에 더 큰 불확실성을 겪는다.

추가 장벽:

- 한국어 중심의 안내
- 지하철 출구와 환승 구조 이해 어려움
- 교통카드 발급 및 충전 어려움
- 해외카드 결제 가능 여부 불확실
- 음식 알레르기, 노키즈존, 예약 문화 이해 어려움
- 영업시간과 브레이크타임 정보 불확실

정리 문장:

> 외국인 관광객은 혼잡 문제와 함께 서울의 로컬 정보 장벽까지 겪기 때문에, 더 쉬운 행동 안내가 필요하다.

## 6. 문제 정의

서울을 방문하는 사용자는 여행지와 이동 경로를 선택할 때 시간뿐 아니라 혼잡도, 날씨, 대기시간, 안전성, 언어 접근성을 함께 고려해야 한다.

하지만 현재 지도 앱과 관광 앱은 이 정보를 통합해 “지금 가도 되는지”와 “어디로 바꾸면 좋은지”를 행동 중심으로 안내하지 못한다.

따라서 사용자는 실시간 혼잡도와 날씨, 대중교통 데이터를 바탕으로 현재 상황을 쉽게 이해하고, AI가 제안하는 실행 가능한 대안을 받을 수 있는 서비스가 필요하다.

## 7. 솔루션 개요

널널은 서울의 혼잡 데이터와 AI 분석을 통해 사용자가 더 널널한 선택을 하도록 돕는다.

### 핵심 가치

- 방문 전 혼잡도를 확인한다.
- 혼잡이 증가할 시간을 예측한다.
- 혼잡하거나 날씨가 나쁘면 대체 장소를 추천한다.
- 숫자가 아니라 행동 중심 메시지를 제공한다.
- 외국인 관광객도 쉽게 이해할 수 있게 안내한다.

### 핵심 컨셉

> 혼잡도 데이터는 보여주는 것에서 끝나면 안 된다.  
> 널널은 AI가 데이터를 해석해 “지금 사용자가 해야 할 행동”으로 바꿔준다.

## 8. 서비스 구조

하단 네비게이션 순서:

```text
지도 → 길찾기 → 홈 → AI비서 → 마이페이지
```

### 화면별 역할

| 화면 | 역할 |
|---|---|
| 지도 | 서울시 주요 지역 밀집도를 샘플로 보여준다 |
| 길찾기 | 현재 위치와 목적지를 선택하고 대중교통 경로를 확인한다 |
| 홈 | 현재 붐비는 장소와 현재 위치 기반 날씨를 보여준다 |
| AI비서 | 목적지 분석, 경고 메시지, 추천 문구를 보여준다 |
| 마이페이지 | 고객센터, 설정, 알림설정 등 기본 기능을 제공한다 |

홈 화면은 서비스의 중심 화면이므로 다른 탭보다 더 크고 강조된 UI로 설계한다.

## 9. 핵심 기능 상세

### 9.1 홈

홈은 사용자가 앱을 켰을 때 가장 먼저 보는 화면이다.

주요 구성:

- 현재 위치 기반 날씨
- 현재 붐비는 장소
- 혼잡 상승 지역 요약
- 지금 대체하기 좋은 장소
- 지도/길찾기로 이동하는 빠른 버튼

사용자 가치:

- 앱을 켜자마자 지금 피해야 할 곳을 알 수 있다.
- 날씨에 따라 실외/실내 코스를 판단할 수 있다.
- 복잡한 정보 탐색 없이 현재 상황을 빠르게 이해할 수 있다.

### 9.2 지도

지도 탭은 서울시 주요 장소의 밀집도를 보여준다.

초기 MVP에서는 서울 실시간 도시데이터 API를 바로 연결하지 못할 수 있으므로, mock 데이터로 주요 장소 밀집도 샘플을 제공한다.

표시 정보:

- 장소명
- 현재 혼잡도
- 현재 추정 인구
- 30분 뒤 예측 변화
- 혼잡도 단계

혼잡도 단계:

| 단계 | 의미 | 사용자 안내 |
|---|---|---|
| 여유 | 이동과 체류가 비교적 편함 | 지금 방문 추천 |
| 보통 | 일반적인 수준 | 방문 가능 |
| 혼잡 | 이동 불편 가능성 있음 | 시간 조정 권장 |
| 매우 혼잡 | 계획 차질 가능성 높음 | 대체 장소 추천 |

### 9.3 길찾기

길찾기 탭은 네이버 지도처럼 현재 위치와 목적지를 선택하는 화면이다.

사용 흐름:

1. 사용자가 현재 위치를 확인한다.
2. 목적지를 입력하거나 선택한다.
3. 선택 완료 버튼을 누른다.
4. 대중교통 이용방법을 확인한다.
5. 목적지 분석 결과 보기 버튼을 통해 AI비서 화면으로 이동한다.

제공 경로:

- 빠른 경로
- 균형 추천 경로
- 덜 붐비는 경로

각 경로에는 다음 정보를 제공한다.

- 예상 이동 시간
- 도보 시간
- 요금
- 혼잡도
- 쾌적 점수
- AI 추천 이유

### 9.4 AI비서

AI비서 탭은 목적지 분석 결과를 보여주는 핵심 화면이다.

경고 메시지가 있는 경우:

- 혼잡 경고
- 예상 위험 시간
- 경고 이유
- 대체 행동
- 대체 장소 추천

경고 메시지가 없는 경우:

- 추천 맛집
- 추천 장소
- AI가 예측한 방문 팁
- 날씨 기반 안내 문구

예시 문구:

```text
길라미 AI가 현재 홍대입구역 9번 출구 주변 혼잡을 감지했어요.
30분 뒤 매우 혼잡할 가능성이 높습니다.
성수 카페거리로 일정을 바꾸면 이동 시간은 12분 늘지만 혼잡도는 낮아집니다.
```

### 9.5 마이페이지

마이페이지는 기본 앱 기능을 제공한다.

포함 기능:

- 회원정보 수정
- 로그아웃
- 회원 탈퇴
- 알림 설정
- 언어 설정
- 위치 권한 설정
- 저장한 장소
- 공지사항
- 고객센터
- 데이터 연결 상태
- 개인정보 및 안전 안내

## 10. 타겟 사용자

### 1차 타겟

서울을 처음 방문하거나 서울 지리에 익숙하지 않은 외국인 관광객.

특징:

- Google Maps, Instagram, TikTok에 의존한다.
- 서울 로컬 정보와 대중교통 구조에 익숙하지 않다.
- 짧은 여행 기간 동안 헛걸음을 줄이고 싶어 한다.
- 한국어 안내를 이해하기 어렵다.

### 2차 타겟

서울 핫플레이스, 팝업스토어, 맛집, 카페를 자주 방문하는 내국인.

특징:

- 인스타그램, X, 틱톡에서 본 장소를 주말에 방문한다.
- 대기줄과 혼잡도를 도착 후에 알게 되는 경우가 많다.
- 몇 시에 가야 덜 붐비는지 알고 싶어 한다.

### 3차 타겟

아이 또는 노약자와 함께 서울에 방문하는 가족 단위 사용자.

특징:

- 사람이 너무 많은 장소를 피하고 싶어 한다.
- 조금 돌아가더라도 넓고 안전한 동선을 선호한다.
- 날씨 변화와 휴식 장소에 민감하다.

## 11. 페르소나

### Persona 1. Emma Johnson

| 항목 | 내용 |
|---|---|
| 이름 | Emma Johnson |
| 나이 | 27세 |
| 국적 | 미국 |
| 유형 | 서울 첫 방문 외국인 관광객 |
| 여행 스타일 | SNS에서 본 핫플레이스 중심 여행 |

핵심 상황:

> 서울에서 갈 곳은 많은데, 막상 도착해보니 사람이 너무 많으면 어디로 바꿔야 할지 모르겠어요.

Pain Points:

- 도착 전까지 혼잡도를 알기 어렵다.
- 한국어 정보와 로컬 상황을 이해하기 어렵다.
- 비가 오거나 사람이 많으면 대체 장소를 찾기 어렵다.
- 숫자보다 “지금 어디로 가야 하는지”가 필요하다.

Needs:

- 출발 전 혼잡도 확인
- 외국어로 된 쉬운 안내
- 비슷한 분위기의 대체 장소 추천
- 지금 가도 되는지에 대한 행동 중심 안내

연결 기능:

| 문제 | 기능 |
|---|---|
| 도착 후에야 혼잡을 알게 됨 | 실시간 혼잡도 지도 |

| 한국어 정보가 어려움 | 영어 우선 다국어 UI |

Pain Points:

Needs:

- 시간대별 혼잡 예측
- 지금 갈지, 나중에 갈지 판단 기준
- 대기 중 갈 수 있는 근처 대체 장소 추천

## 13. 데이터 및 API 전략

### MVP 데이터 구성

초기에는 API 키와 실시간 데이터 연결이 완전하지 않을 수 있으므로, mock 데이터로 전체 사용 흐름을 먼저 구현한다.

### 필요한 API

| API | 제공처 | 용도 | 비용 | 초기 대응 |
|---|---|---|---|---|
| 서울 실시간 도시데이터 | 서울열린데이터광장 | 핵심 인구 데이터 | 무료 | mock → 실제 API 전환 |
| 서울교통공사 승하차 인원 | 공공데이터포털 | 지하철 혼잡 proxy | 무료 | 시간대별 패턴 보정 |
| 기상청 단기예보 | 공공데이터포털 | 강수확률, 기온 | 무료 | 날씨 기반 추천 |
| ODsay 대중교통 길찾기 | 아로정보기술 | 대중교통 경로 | 무료 구간 있음 | 길찾기 후보 경로 생성 |

### 데이터 처리 방식

1. 서울 실시간 도시데이터로 장소별 인구밀집도를 가져온다.
2. 지하철 승하차 데이터를 이용해 현재 이동량 변화를 보정한다.
3. 날씨 데이터를 활용해 실외/실내 장소 추천 기준을 조정한다.
4. ODsay API로 대중교통 경로를 생성한다.
5. AI가 혼잡도, 이동시간, 날씨, 사용자 목적지를 종합해 추천 문구를 생성한다.

### 보안 원칙

- API 키는 앱에 직접 넣지 않는다.
- 백엔드 프록시 서버에서 API를 호출한다.
- 앱은 정규화된 응답만 받는다.
- 위치 정보는 사용자의 동의가 있을 때만 사용한다.

## 17. 사용자 스토리와 수용 기준

### Story 1. 현재 붐비는 장소 확인

As a 서울 방문자, I want to 현재 붐비는 장소를 확인하고 싶다, so that 헛걸음을 줄일 수 있다.

수용 기준:

1. 홈 화면에서 현재 붐비는 장소 목록이 보인다.
2. 각 장소에는 혼잡도 단계가 표시된다.
3. 각 장소에는 현재 추정 인구와 예측 변화가 표시된다.
4. 장소를 누르면 상세 화면으로 이동한다.

### Story 2. 목적지 길찾기

As a 관광객, I want to 현재 위치와 목적지를 선택하고 싶다, so that 이동 방법을 쉽게 확인할 수 있다.

수용 기준:

1. 길찾기 화면에서 현재 위치와 목적지를 입력할 수 있다.
2. 선택 완료 버튼을 누르면 경로 후보가 표시된다.
3. 경로 후보는 빠른 경로, 균형 경로, 덜 붐비는 경로로 나뉜다.
4. 각 경로에는 소요시간, 도보시간, 혼잡도, 쾌적점수가 표시된다.
5. 목적지 분석 결과 보기 버튼을 누르면 AI비서 화면으로 이동한다.

### Story 3. AI 경고 메시지 확인

As a 사용자, I want to 목적지의 혼잡 경고를 받고 싶다, so that 지금 갈지 바꿀지 판단할 수 있다.

수용 기준:

1. AI비서 화면에서 목적지 분석 결과가 보인다.
2. 혼잡 위험이 있으면 경고 메시지가 표시된다.
3. 경고 메시지에는 이유와 대체 행동이 포함된다.
4. 대체 장소 카드가 함께 제공된다.
5. 경고가 없으면 추천 맛집이나 방문 팁이 표시된다.

### Story 4. 외국인 친화 안내

As an 외국인 관광객, I want to 쉬운 언어로 안내를 받고 싶다, so that 서울 이동을 스스로 판단할 수 있다.

수용 기준:

1. 주요 안내 문구는 쉬운 한국어와 영어 전환을 고려해 작성된다.
2. 지하철 출구, 대체 장소, 경고 이유가 짧고 명확하게 표시된다.
3. 숫자만 보여주지 않고 행동 중심 메시지를 제공한다.

## 18. 성공 지표

### North Star Metric

**AI 추천으로 계획 차질을 피한 세션 수**

정의:

사용자가 혼잡 경고, 대체 장소 추천, 덜 붐비는 경로 중 하나를 확인하고 실제로 목적지나 경로를 변경한 세션 수.

|---|---|

| 덜 붐비는 경로 선택률 | 사용자가 빠른 길보다 쾌적한 길을 선택하는지 확인 |
| 재방문율 | 여행 중 반복 사용 가능성 확인 |
| 외국인 사용자 만족도 | 주 타겟의 체감 가치 확인 |

## 20. 수익 모델

### 1차 수익 모델

AI 플랜 기능을 부분 유료화한다.

예시:

- AI 플랜 3회 무료
- 이후 토큰 결제
- 여행 일정 생성, 경로 재조정, 혼잡 회피 플랜을 유료 기능으로 제공

### 2차 수익 모델

상인과 관광객을 연결하는 상권 추천 모델을 도입한다.

가능한 방식:

- 상인이 장소 등록 시 1회성 등록비 결제
- 추천 노출 광고
- 예약 또는 주문 전환 시 수수료
- 혼잡 지역 주변 대체 상권 프로모션

주의점:

- 추천 신뢰도를 해치지 않기 위해 광고와 AI 추천은 명확히 구분해야 한다.
- 초기에는 수익화보다 사용자 신뢰 확보가 우선이다.

## 21. 제품 전략

### 시장 포지션

널널은 지도 앱을 대체하는 서비스가 아니다. 지도 앱 위에 얹히는 **혼잡 회피와 행동 추천 특화 레이어**다.

### 우리가 집중할 것

- 서울 주요 핫플레이스
- 외국인 관광객의 현장 의사결정
- 혼잡 회피
- 날씨 기반 대체 추천
- 행동 중심 AI 안내

### 하지 않을 것

- 모든 도시를 한 번에 지원하지 않는다.
- 모든 장소 리뷰 플랫폼이 되지 않는다.
- 일반 맛집 앱처럼 리뷰 수 경쟁을 하지 않는다.
- 초기부터 완전한 여행 예약 플랫폼을 만들지 않는다.
- 공포를 유발하는 위험 경고 서비스로 포지셔닝하지 않는다.

## 22. 기술 아키텍처

### 22.1 아키텍처 목표

널널의 기술 구조는 초기 MVP에서는 mock 데이터로 빠르게 검증하고, 이후 실제 공공데이터와 AI 분석을 안정적으로 연결할 수 있도록 설계한다.

핵심 목표:

- API 키를 앱에 직접 노출하지 않는다.
- 공공데이터, 날씨, 길찾기 데이터를 한 곳에서 정규화한다.
- 데이터 지연이나 API 장애가 있어도 앱이 멈추지 않도록 fallback을 둔다.
- AI가 숫자 데이터를 행동 가능한 문구로 바꿀 수 있게 한다.
- 외국인 관광객에게 쉬운 다국어 안내를 제공할 수 있게 확장한다.

### 22.2 전체 시스템 구조

```text
사용자 앱
  ↓
Frontend App
  - React Native / Expo
  - 지도, 길찾기, 홈, AI비서, 마이페이지
  - 현재는 mock JSON 데이터 사용
  ↓
Backend API Gateway
  - 클라이언트 요청 수신
  - 인증/권한 처리
  - API 키 보호
  - 데이터 정규화 응답 제공
  ↓
Data Adapter Layer
  - 서울 실시간 도시데이터 Adapter
  - 서울교통공사 승하차 데이터 Adapter
  - 기상청 단기예보 Adapter
  - ODsay 대중교통 길찾기 Adapter
  ↓
Data Processing Layer
  - 장소별 혼잡도 정규화
  - 시간대별 혼잡 예측
  - 날씨 기반 추천 조건 계산
  - 경로별 쾌적 점수 계산
  ↓
AI Recommendation Layer
  - 길라미 AI 메시지 생성
  - 경고 문구 생성
  - 대체 장소 추천
  - 다국어 안내 문구 생성
  ↓
Database / Cache
  - 장소 정보
  - 혼잡도 snapshot
  - 예측 결과
  - 사용자 저장 장소
  - 알림 설정
```

### 22.3 프론트엔드 구조

기술 스택:

- React Native
- Expo Router
- TypeScript
- React Native Chart Kit
- mock JSON 기반 데이터 시뮬레이션

주요 화면:

| 화면 | 역할 | 주요 데이터 |
|---|---|---|
| 지도 | 서울 주요 지역 밀집도 표시 | 장소별 혼잡도, 현재 인구 |
| 길찾기 | 현재 위치/목적지 선택, 경로 비교 | 경로 후보, 이동시간, 혼잡도 |
| 홈 | 현재 붐비는 장소와 날씨 요약 | 혼잡 TOP, 날씨, 추천 장소 |
| AI비서 | 목적지 분석과 경고 메시지 | AI 분석 결과, 대체 장소 |
| 마이페이지 | 설정, 알림, 고객센터 | 사용자 설정, 데이터 상태 |

프론트엔드 설계 원칙:

- 사용자는 숫자보다 행동 안내를 먼저 본다.
- 혼잡도는 색상과 짧은 문구로 빠르게 이해하게 한다.
- 외국인도 이해할 수 있도록 짧은 문장과 명확한 버튼을 사용한다.
- API 장애 시에도 mock/fallback 데이터로 화면이 깨지지 않게 한다.

### 22.4 백엔드 API Gateway

백엔드는 앱과 외부 API 사이의 중간 서버 역할을 한다.

주요 역할:

- 공공데이터 API 키 보호
- ODsay API 키 보호
- 외부 API 응답 형식 정규화
- 캐시된 데이터 제공
- AI 분석 요청 관리
- 사용자 설정 저장

예상 엔드포인트:

| Method | Endpoint | 설명 |
|---|---|---|
| GET | `/api/places` | 주요 장소 목록 조회 |
| GET | `/api/places/:id/crowd` | 특정 장소 혼잡도 조회 |
| GET | `/api/weather/current` | 현재 위치 기반 날씨 조회 |
| POST | `/api/routes/search` | 출발지/목적지 기반 경로 조회 |
| POST | `/api/ai/analyze-destination` | 목적지 AI 분석 요청 |
| GET | `/api/recommendations/alternatives` | 대체 장소 추천 조회 |
| POST | `/api/notifications/settings` | 알림 설정 저장 |

### 22.5 데이터 Adapter 설계

외부 API는 제공처마다 응답 형식, 지연 시간, 제한 조건이 다르다. 따라서 직접 화면에서 API를 호출하지 않고, Adapter 계층에서 같은 형식으로 변환한다.

| Adapter | 입력 | 출력 |
|---|---|---|
| Seoul CityData Adapter | 장소 코드 | 현재 인구, 혼잡 단계, 업데이트 시각 |
| Subway Adapter | 역 코드, 시간대 | 승하차 추정값, 시간대 패턴 |
| Weather Adapter | 좌표 | 강수확률, 기온, 날씨 상태 |
| ODsay Adapter | 출발/도착 좌표 | 대중교통 경로 후보 |

정규화 예시:

```json
{
  "placeId": "hongdae",
  "name": "홍대입구",
  "crowdLevel": "매우 혼잡",
  "population": 62000,
  "forecastPopulation": 72000,
  "updatedAt": "2026-05-11T18:20:00+09:00",
  "source": "seoul-citydata"
}
```

### 22.6 AI Recommendation Layer

길라미 AI는 데이터 자체를 생성하는 것이 아니라, 여러 데이터를 해석해 사용자가 바로 이해할 수 있는 문구로 바꾼다.

입력 데이터:

- 목적지
- 현재 위치
- 도착 예정 시간
- 현재 혼잡도
- 예측 혼잡도
- 날씨
- 경로 후보
- 대체 장소 후보
- 사용자 언어

출력 데이터:

- 경고 여부
- 경고 메시지
- 추천 행동
- 추천 이유
- 대체 장소
- 예상 추가 이동시간
- 혼잡 완화 기대치

AI 메시지 원칙:

- 공포를 조장하지 않는다.
- 확정 표현보다 가능성 표현을 사용한다.
- 수치보다 행동을 먼저 말한다.
- 추천 이유를 함께 제공한다.
- 외국인에게 어려운 로컬 표현은 쉽게 풀어 쓴다.

예시:

```text
길라미 AI가 홍대입구역 9번 출구 주변 혼잡 상승을 감지했어요.
30분 뒤 매우 혼잡할 가능성이 높습니다.
성수 카페거리로 이동하면 12분 더 걸리지만, 혼잡도는 낮아질 것으로 예상됩니다.
```

### 22.7 데이터 저장소 설계

MVP에서는 간단한 구조로 시작하고, 이후 서비스 확장에 맞춰 분리한다.

초기 추천:

- PostgreSQL: 장소, 사용자, 저장 장소, 알림 설정
- Redis: 실시간 혼잡도 snapshot, API 캐시, 예측 결과 캐시
- Object Storage: 로그, 분석 리포트, 장기 데이터 백업

주요 테이블:

| 테이블 | 설명 |
|---|---|
| `places` | 장소 기본 정보 |
| `crowd_snapshots` | 장소별 혼잡도 기록 |
| `weather_snapshots` | 위치별 날씨 기록 |
| `route_results` | 경로 탐색 결과 |
| `ai_recommendations` | AI 분석 결과 |
| `users` | 사용자 정보 |
| `saved_places` | 저장한 장소 |
| `notification_settings` | 알림 설정 |

### 22.8 혼잡 예측 로직

MVP에서는 복잡한 머신러닝보다 규칙 기반 점수화로 시작한다.

혼잡 위험 점수 예시:

```text
혼잡 위험 점수 =
현재 혼잡도 점수
+ 30분 뒤 인구 증가율 점수
+ 지하철 승하차 증가 점수
+ 행사/팝업 여부 점수
+ 날씨 영향 점수
```

점수 단계:

| 점수 | 단계 | 안내 |
|---|---|---|
| 0~30 | 여유 | 지금 방문 추천 |
| 31~55 | 보통 | 방문 가능 |
| 56~75 | 혼잡 | 시간 조정 권장 |
| 76~100 | 매우 혼잡 | 대체 장소 추천 |

향후에는 실제 사용 로그와 공공데이터 이력을 활용해 예측 모델을 고도화한다.

### 22.9 알림 구조

알림은 사용자가 저장한 장소, 선택한 목적지, 일정에 따라 발송한다.

알림 조건:

- 목적지 혼잡도가 “매우 혼잡”으로 상승 예상
- 비/폭염/한파로 실외 장소 만족도 하락 예상
- 행사 종료로 특정 역 승하차 급증 예상
- 사용자가 저장한 장소의 혼잡도 급상승

알림 예시:

```text
홍대입구가 30분 뒤 매우 혼잡해질 수 있어요.
지금 출발한다면 2번 출구를 이용하거나 성수 카페거리로 변경해보세요.
```

### 22.10 장애 대응 및 Fallback

| 장애 상황 | 대응 |
|---|---|
| 서울 실시간 도시데이터 API 실패 | 마지막 캐시 데이터와 업데이트 시각 표시 |
| 날씨 API 실패 | 최근 날씨 캐시 사용 |
| ODsay API 실패 | mock 경로 또는 기본 경로 안내 |
| AI 분석 실패 | 규칙 기반 메시지 fallback |
| 위치 권한 거부 | 사용자가 직접 지역 선택 |
| 데이터 지연 | “공공데이터 지연값 + AI 보정값”으로 표시 |

### 22.11 보안 및 개인정보 원칙

- API 키는 앱에 저장하지 않는다.
- 위치 정보는 사용자 동의 후에만 사용한다.
- 위치 정보는 필요한 기간만 보관한다.
- 민감한 개인 이동 기록은 익명화한다.
- 외부 API 호출은 백엔드에서 관리한다.
- AI 추천 로그는 서비스 개선 목적임을 명확히 고지한다.

### 22.12 MVP 구현 순서

1. mock JSON 기반 화면 완성
2. 백엔드 API 응답 스키마 정의
3. 서울 실시간 도시데이터 Adapter 구현
4. 날씨 Adapter 구현
5. ODsay 길찾기 Adapter 구현
6. AI 분석 API 구현
7. 캐시와 fallback 적용
8. 알림 설정과 저장 장소 구현
9. 다국어 문구 확장

## 23. 로드맵

### Phase 1. MVP 검증

목표:

외국인 관광객과 핫플 방문자가 혼잡도와 대체 장소 추천을 실제로 유용하게 느끼는지 검증한다.

주요 기능:

- 홈
- 지도
- 길찾기
- AI비서
- 마이페이지
- mock 데이터 기반 시뮬레이션

성공 기준:

- 사용자 20명 이상 테스트
- “이 앱이 있으면 사용할 것 같다” 응답 60% 이상
- AI 추천 이해도 70% 이상

### Phase 2. 실제 API 연동

목표:

mock 데이터를 실제 공공데이터와 날씨, 교통 API로 대체한다.

주요 기능:

- 서울 실시간 도시데이터 연동
- 기상청 단기예보 연동
- 대중교통 길찾기 API 연동
- 데이터 업데이트 시각 표시

성공 기준:

- 주요 장소 7곳 이상 실제 데이터 반영
- 예측 메시지 생성 자동화
- 데이터 오류 fallback 설계

### Phase 3. AI 플랜 확장

목표:

사용자가 “2박 3일 홍대 데이트 코스”처럼 자연어로 요청하면 AI가 혼잡도와 날씨를 고려해 일정을 짜준다.

주요 기능:

- AI 일정 생성
- 일정 저장
- 혼잡 변화 시 일정 재조정
- 유료 토큰 모델 테스트

성공 기준:

- AI 플랜 생성 후 저장률 30% 이상
- 유료 전환 의향 조사 20% 이상

### Phase 4. 상권 연계

목표:

혼잡 지역의 방문 수요를 주변 상권으로 분산하고, 상인과 관광객을 연결한다.

주요 기능:

- 상인 장소 등록
- 추천 노출
- 프로모션
- 예약/주문 연계

성공 기준:

- 제휴 상점 30곳 이상
- 대체 장소 클릭 후 방문 의향 40% 이상

## 25. 최종 포지셔닝 문장

> 널널은 서울을 방문하는 사람이 혼잡 때문에 헛걸음하지 않도록, AI가 실시간 혼잡도와 날씨, 이동 정보를 분석해 지금 가도 되는 장소와 더 나은 대안을 알려주는 여행 비서 서비스다.

---

## 26. 발표용 핵심 문장

- AI가 헛걸음을 없앤다면, 서울 여행은 더 널널해질 수 있다.
- 널널은 숫자가 아니라 행동을 추천한다.
- 사용자는 “몇 명이 있나”보다 “지금 가도 되나”를 알고 싶어 한다.
- 혼잡한 핫플 대신, 더 널널한 대안을 제안한다.
- 외국인 관광객의 서울 여행 실패를 줄이는 AI 여행 비서다.
